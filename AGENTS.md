# AGENTS Instruction

## Big picture

- This is a single-page, vanilla JS P2P shooter (no build step). The entry point is `index.html`, which loads scripts in order: `constants.js` -> `sound.js` -> `physics.js` -> `renderer.js` -> `network.js` -> `game.js`.
- Core loop and state live in the `Game` module (IIFE). `Game` owns the tick, input handling, mode switching (local/online host/online guest), and state transitions.
- Networking is host-authoritative. The host runs the canonical physics simulation. **P1 (host's player)**: the host sends its input keys **and authoritative P1 position** to the guest at **60 Hz** (`host_input` message). The guest runs `processRemoteInput(p1)` for responsive bullet creation only; the rendered P1 position is **directly applied** from `hostP1State` every frame (not gated by physics tick). **P2 (guest's player)**: the guest sends its input to the host at **60 Hz** (`input` message); the host processes P2's movement authoritatively and sends back P2's position/direction at **60 Hz** inside the `host_input` message. The guest applies P2 positions directly from `hostP2State` every frame — a "dumb terminal" rendering model that eliminates jitter by simply displaying whatever the host last sent. The guest still handles P2 shooting locally for responsive bullet creation. The host also sends lightweight **authority corrections** at **10 Hz** (`correction` message) containing player health/scores, both player positions, bombs, zombies, explosions, and game state; correction positions also update `hostP1State`/`hostP2State`. Authority state (health, score, alive, game state) is snapped immediately. Bomb/zombie **spawning** only happens on the host; the guest ticks existing entities but receives new ones via corrections. Full state (`broadcastState`) is only sent once after reconnection and also seeds `hostP1State`/`hostP2State`. See `sendHostInput()`/`sendCorrections()`/`applyCorrections()` in `game.js`.
- **Render interpolation** — the game loop uses a fixed 60 Hz physics timestep with `requestAnimationFrame` for rendering. On the **host/local** side, player and bullet positions are interpolated between the previous and current physics state using `alpha = physicsAccumulator / PHYSICS_STEP_MS` to avoid micro-stutter on high-refresh displays. `savePhysicsPositions()` snapshots `prevX/prevY` before each physics step; `interpolateForRender(alpha)` temporarily overwrites entity positions for drawing; `restorePhysicsPositions()` puts them back after rendering. On the **guest** side, player positions are **not** render-interpolated — they are applied directly from host state every frame (the "dumb terminal" model), so save/interpolate/restore only applies to **bullets**. Bombs, explosions, and zombies don't move so they are not interpolated.

## Architecture and data flow

- `constants.js` defines gameplay constants, maze data, and `parseMaze()`. It also owns the global `activeMaze` used by `Physics` and `Renderer`.
- `physics.js` uses `activeMaze.walls` for collision checks; keep wall rectangles aligned to the maze grid.
- `renderer.js` renders the maze grid (`activeMaze.grid`) and HUD every frame. It assumes the retro color palette in `COLORS` from `constants.js`.
- `network.js` wraps PeerJS (via CDN) and manages room codes, connection lifecycle, and the host/guest roles. It attaches a `peer.on('disconnected')` handler on every peer that calls `peer.reconnect()` automatically when the signaling WebSocket drops (e.g. mobile backgrounding). `restoreHost(roomCode, cbs)` and `restoreGuest(roomCode, cbs)` **reuse the existing Peer when possible** — `restoreHost` checks `peer.id === expectedId && !peer.destroyed` and, if true, just calls `peer.reconnect()` (for a dropped signaling WS) or fires `onReady` immediately (if the peer is already open) instead of destroying and creating a new Peer; `restoreGuest` similarly reuses the existing Peer and calls `peer.connect()` directly when `peer.open`, or `peer.reconnect()` when the signaling WS dropped. A new Peer is only created when the old one is destroyed or has the wrong ID. This avoids `unavailable-id` errors from the PeerJS server’s ~60 s peer ID TTL and eliminates redundant signaling WebSocket connections on every retry. All event handlers attached during original peer creation (open, connection, error) reference the module-level `callbacks` object so they stay correct after `callbacks = cbs` is updated in each restore call. `getLastRoomCode()` / `setLastRoomCode(code)` expose/set the saved code. `generateRoomCode()` is public so `game.js` can pre-generate a code before registering the peer. `createHostWithCode(roomCode, cbs)` creates a host peer with an explicit code (used in Phase 2 of deferred room creation). Key internal invariants: (1) `setupConnection(connRef)` and `startOpenPoll(connRef)` take an explicit `DataConnection` reference rather than reading the module-level `conn`, preventing stale references when `conn` is replaced during reconnects; (2) all paths that call `markConnected()` guard with `connRef === conn` before firing; `startOpenPoll` also cancels itself immediately when `connRef !== conn`; (3) `close`/`error` handlers guard with `connRef === conn` before firing callbacks, so events from replaced connections are silently dropped; (4) the `intentionallyDestroyedPeers` **Set** (replacing the old boolean `intentionalDestroy` flag) suppresses `onPeerClosed` when `peer.destroy()` is called deliberately; the peer reference is added to the Set before calling `destroy()` and removed when its `close` event fires, avoiding the async timing race of a reset-before-fire boolean; additionally `attachPeerLifecycleHandlers` guards the `close` handler with `peerObj !== peer` to drop late close events from replaced peer instances; (5) `disconnect()` resets `isHost` and `lastRoomCode` to avoid leaking stale state into the next session; (6) `PEER_DEBUG` is `2` on `localhost` and `0` everywhere else; (7) ICE servers are limited to the 5 known-good Google STUN servers — dead community VoIP STUN servers have been removed to reduce ICE gathering latency.
- `sound.js` provides procedural audio via Web Audio API (IIFE, `Sound` global). All 11 sound effects are synthesized — zero audio files. Both host and guest generate sounds locally from their own simulation (`Sound.play()` calls in shared physics functions like `tryShoot`, `handlePlayerDeath`, `explodeBomb`, etc.). On reconnection, the one-shot `broadcastState` still includes `sounds` via `Sound.flushPending()` for backward compatibility. Stereo panning is position-based (event x-coordinate mapped to left/right speaker). Mute state is persisted in `localStorage('p2p-muted')`.
- `index.html` contains the lobby UI and inline script for building the maze selector; it calls `Game.*` entry points directly.
- **Mobile touch controls** — `#touch-controls` (inside `#gameContainer`) is a full-width bar shown only under `@media (orientation: portrait) and (max-width: 1024px)`. Left half: `#shoot-btn` (maps `touchstart/end` → `keys["Space"]`). Right half: `#joystick-zone` with a floating-knob joystick that maps 8 directional sectors → `keys["KeyW/A/S/D"]`. Wired in `initTouchControls()` in `game.js`, called from `init()`. `resizeCanvas()` subtracts `#touch-controls.offsetHeight` so the canvas never overlaps the controls bar.

## Project-specific conventions

- Modules are plain IIFEs in the global scope (no bundler, no imports). Keep new code consistent with this pattern.
- `activeMaze` is a shared global; when changing mazes call `parseMaze()` and update `selectedMazeKey` so Renderer/Physics stay in sync.
- Online guest uses WASD + Space for input (same as host). Local 2-player uses WASD/Space for P1 and arrows/Enter for P2.
- Maze rotation happens every 1 minute; all 6 mazes are shuffled and played in order. After all 6 mazes (6 minutes total), the match ends and the player with the most kills wins. A player can also win instantly by reaching `WIN_SCORE` (5) kills. If scores are tied at timeout, health is used as a tiebreaker; otherwise it's a draw.
- **Immediate room creation** — tapping "CREATE GAME ROOM" calls `showOnlineUI('host')` which calls `Network.generateRoomCode()`, displays the room code and share link, and **immediately** registers with PeerJS via `setupHostRoom(roomCode, true)`. There is no longer a deferred Phase 2 — the peer is created right away.
- **Lobby retry limit** — `setupHostRoom` tracks retries via `lobbyRetryCount` (module-level). After `LOBBY_MAX_RETRIES = 5` failures, an error message is shown and the retry loop stops — the user can press Cancel to return to the lobby. The count is reset on success (`onConnected`), on `showOnlineUI('host')`, on `cancelOnline()`, and on `returnToLobby()`.
- **Ready handshake** — after the DataChannel opens, the guest immediately sends `{ type: "ready" }`. The host's `setupHostRoom.onData` intercepts this and only then calls `startOnlineGame(true)` + sends `{ type: "config" }`. This prevents the race where the host sends config before the guest's `onData` handler is wired. If the host is **already mid-game** when it receives "ready" (e.g. the guest refreshed or joined late), `handleNetworkData` catches it and responds with `config` + `resync` + `broadcastState(true)` to bootstrap the guest into the running match — config starts the guest's game loop, resync corrects maze/timer state, and broadcastState syncs all positions/scores.
- **Resync vs config** — mid-game reconnection uses a new `{ type: "resync" }` message instead of `{ type: "config" }`. The `resync` message syncs `mazeKey`, `mazeOrder`, `mazesPlayed`, `matchStartTime`, and `mazeRotationStart` without resetting scores or restarting the match. `handleNetworkData` handles both types.
- **Mobile reconnection** — `STATE.RECONNECTING` is entered when the DataChannel closes but the game is still in `online-host`/`online-guest` mode. `game.js` runs a **75 s** reconnect window (increased from 30 s to outlast PeerJS server’s ~60 s peer ID expiry TTL): every 3 s it calls `Network.restoreHost` or `Network.restoreGuest` which **reuse the existing Peer** (only closing the DataConnection, not the Peer itself) so there are no redundant signaling WebSocket connections or `unavailable-id` errors. A `visibilitychange` listener on `document` triggers the same flow proactively when the tab comes back to the foreground; it also handles the lobby phase — if the peer died while waiting for a guest, it re-registers the room with the same code. On success the host re-sends `resync` + a full state broadcast. On timeout, `handleDisconnect` falls through to the normal lobby return. The amber `drawReconnecting(secondsLeft)` overlay in `renderer.js` shows a countdown during this window.
- **Guest data-timeout detection** — `lastDataReceived` is updated in `handleNetworkData`. If the guest is in active play and receives no data for `DATA_TIMEOUT_MS = 5000` ms, `handleDisconnect()` is triggered automatically (covers silent host crashes / network drops with no WebRTC close event). `lastDataReceived` is reset to `Date.now()` in `startOnlineGame` and to `0` after a timeout trigger to prevent repeated triggers.
- **Session persistence (page reload)** — `startOnlineGame()` saves `{ role, roomCode }` to `sessionStorage` under key `p2p-maze-shooter`. On page load (`DOMContentLoaded` in `index.html`), `Game.rejoinSession()` reads this and auto-creates the host room with the same code or auto-joins as guest — before checking for `?room=` URL params. The session is cleared in `returnToLobby()`, `cancelOnline()`, and on hard join errors so it doesn't loop. This lets both host and guest survive a full page reload: the host re-registers the same peer ID and the guest reconnects to it. **Game state preservation** — `sendCorrections()` (10 Hz) also calls `saveSession` with the full game state (`mazeOrder`, `selectedMazeKey`, `mazesPlayed`, `matchStartTime`, `mazeRotationStart`, `p1Score`, `p2Score`). `rejoinSession()` for the host stores this as `pendingResumeState` before calling `setupHostRoom`. `startOnlineGame()` checks `pendingResumeState` when `isHost`: if set, it restores the saved maze/score state and sets `gameState = STATE.PLAYING` directly (skipping the countdown). Immediately after `config`, `setupHostRoom.onData "ready"` always follows with `resync` + `broadcastState(true)` so the guest is also bootstrapped with the restored state.

## Workflows

- No build system or tests. Open `index.html` directly in a browser (or serve statically if needed).
- PeerJS is loaded from `https://cdn.jsdelivr.net/...` in `index.html`; changes to networking should account for PeerJS behavior and events in `network.js`.

## Rules

- Whenever agentic coding make changes that require to update `AGENTS.md`, then update the instruction.

## Where to look

- Game loop + state machine: `game.js`
- PeerJS wrapper and room code handling: `network.js`
- Procedural sound effects: `sound.js`
- Collision rules: `physics.js`
- Rendering and HUD: `renderer.js`
- Maze definitions and parsing: `constants.js`
- Lobby and connection UI: `index.html` + `styles.css`
